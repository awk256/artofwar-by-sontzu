
# 概要

孫子の「五事」は、どれか一つが欠けても致命傷になります。ここでは、現代のシステム開発やAI導入において「五事」が崩れた際に発生する、典型的なアンチパターンを整理します。

# 1. 道（Why）が崩れた失敗：空疎なスローガン

* **症状：** プロジェクトの目的が会議や資料ごとに異なり、メンバーが「自分事」として捉えていない。
* **典型例：** 「全社DX推進」という号令のもと、現場のワークフローを無視したツール導入を行い、結果として**現場の入力負担だけが増える**ケース。
* **結末：** メンバーは「静かな退職（形だけの追従）」を選び、プロジェクトは形骸化します。

# 2. 天（When）が崩れた失敗：早すぎた最適解

* **症状：** 技術的難易度は高いが、市場のニーズやユーザーのITリテラシーが追いついていない。
* **典型例：** LLM（大規模言語モデル）の精度が不十分な時期に、誤りが許されない業務（法務や医療の最終判断など）へ強引に導入し、事故を起こす。
* **結末：** **「技術は良かったが、時期が悪かった」** という総括と共に、多額の投資がサンクコスト化します。

>サンクコスト（埋没費用）とは、すでに投下してしまい、事業や行為を中止しても戻ってこない金銭・時間・労力のこと。

# 3. 地（Where）が崩れた失敗：強者の土俵での死闘

* **症状：** 差別化要因がなく、資本力や規模で勝る競合と同じ土俵で戦っている。
* **典型例：** Big Techが無料で提供している機能と真っ向から競合するSaaSを開発し、資本力による価格競争に巻き込まれる。
* **結末：** 利益率が極限まで低下し、**構造的な赤字**から抜け出せなくなります。

# 4. 将（Who）が崩れた失敗：決断なき迷走

* **症状：** 誰も最終的な責任を取らず、合意形成という名の「意思決定の先送り」が常態化している。
* **典型例：** トラブル発生時にリーダーが現場に責任を押し付けたり、リスクを恐れて技術的な挑戦を土壇場でキャンセルしたりする。
* **結末：** **「この人の下では働けない」** と、組織を支えていた有能なシニアエンジニアから順に去っていきます。

# 5. 法（How）が崩れた失敗：ルールという名の足枷

* **症状：** 現場の実態と評価制度・運用ルールが乖離し、スピードが著しく低下している。
* **典型例：** 「アジャイル開発」を標榜しながら、予算承認やリリース判定には従来型の重厚なウォーターフォール型プロセス（例外処理のオンパレード）を強制する。
* **結末：** 官僚的な調整作業にリソースが削られ、**「仕組みが価値を生む」どころか、仕組みが足を引っ張る**状態になります。

# アーキテクトの警告：ドミノ倒しの構造

孫子の視点に立てば、五事は独立した要素ではなく、連鎖する構造を持っています。

1. **道（Why）** がないから、**将（Who）** が迷う。
2. **将（Who）** が迷うから、**法（How）** が形骸化する。
3. **法（How）** が機能しないから、**天（When）** や **地（Where）** の変化に対応できない。

> **最重要ポイント：**
> 失敗の多くは「技術（How）」の不足ではなく、「道（Why）」や「将（Who）」といった**上位レイヤーの崩壊**から始まります。
