# 1. 【道：Why】導入の必然性と大義

> AIという強力な武器を、なぜ「この戦場」に投入するのかを記述します。

* **背景と目的：** 現在の業務におけるボトルネック、および解決すべき具体的な課題。

# 2. 【将：Who】意思決定と責任の体制

> AIという「自律的な兵」を、誰が御すのかを明確にします。

* **プロジェクト責任者（将）：** [氏名・役職]
* **最終判断権限：** 異常発生時、あるいはAIの出力が疑わしい場合に、利用停止や最終判断を下す責任者は誰か。
* **現場の役割分担：** AIの回答をチェックする担当者、およびフィードバックを行う担当者。

# 3. 【天：When】時機と外部環境の適合性

> 今、この技術を採用することが「早すぎず、遅すぎない」理由を記述します。

* **技術的成熟度：** 採用予定のモデル（LLM等）やツールが実用域にあるか。
* **市場・規制動向：** 関連する法規制（個人情報保護、AI倫理ガイドライン等）への適合状況。
* **コスト妥当性：** 導入コストおよびランニングコストが、現在の市場相場に照らして適正か。


# 4. 【地：Where】データ資産と現場の受容性

> 泥沼（PoC死）を避けるため、自社の足元の状況を客観的に評価します。

* **データの量と質：** 学習や推論に必要なデータが社内に揃っているか。クレンジングの工数は見積もられているか。
* **現場の適合性：** 実際の利用者がAIツールを使いこなせるリテラシーを持っているか。既存フローに無理なく組み込めるか。


# 5. 【法：How】運用規律と安全装置

> 「使いっぱなし」を防ぎ、不敗の形を維持する仕組みを記述します。

* **HITL（Human-in-the-Loop）設計：** AIの出力を人間が検証するプロセス（承認フロー等）の有無。
* **キルスイッチの定義：** どのような状態（例：ハルシネーション率が〇%を超えた等）になったらシステムを止めるのか。
* **監査と改善サイクル：** 定期的な精度チェック、およびログに基づいた再学習の頻度。

# SAからの活用アドバイス

この検討書を埋める過程で、もし **「筆が止まる項目」** があれば、そこがそのプロジェクトの** 「急所（弱点）」**です。

* 例えば「将」が決まらないなら、組織が責任を負う覚悟ができていません。
* 「法」が書けないなら、導入後にトラブルで現場が炎上します。

> **「綺麗な書類を作ること」が目的ではなく、「埋められない項目を事前に見つけ出し、対策を練ること」が、孫子の説く「計（計算）」の本質です。**

>「SAとは、システムアナリスト（Systems Analyst）またはシステムアーキテクト（System Architect）。
